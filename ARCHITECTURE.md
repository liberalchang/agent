# 架构说明（OpenClaw × RAGFlow）

## 一、架构分层总览

```
┌─────────────────────────────────────────────────────┐
│                    总览层                            │
│           README.md / ARCHITECTURE.md               │
├─────────────────────────────────────────────────────┤
│                    身份层                            │
│         SOUL.md / IDENTITY.md / USER.md             │
├─────────────────────────────────────────────────────┤
│                    行为层                            │
│           AGENTS.md / HEARTBEAT.md                  │
├─────────────────────────────────────────────────────┤
│                    记忆层                            │
│          MEMORY.md / memory/*.md                    │
├─────────────────────────────────────────────────────┤
│                    环境层                            │
│                  TOOLS.md                           │
├─────────────────────────────────────────────────────┤
│                    引导层                            │
│                BOOTSTRAP.md                         │
├─────────────────────────────────────────────────────┤
│               知识增强治理层                         │
│    RAGFLOW_POLICY.md / FAQ.md / TERMS.md            │
└─────────────────────────────────────────────────────┘
```

## 二、各层职责

### 1. 身份层

职责：定义 Agent 是谁、如何表达、为谁服务。

### 2. 行为层

职责：定义行为边界、决策优先级、高风险处理原则、周期性维护任务。

### 3. 记忆层

职责：保留长期稳定信息 + 每日交互上下文。

### 4. 环境层

职责：定义工具与外部依赖、接入约束、故障处理。

### 5. 引导层

职责：规定启动顺序、上下文恢复方法、RAGFlow 检查方式、工作模式切换。

### 6. 知识增强治理层

职责：统一检索策略、高频问题口径、术语概念。

## 三、RAGFlow 在整体架构中的位置

```
用户问题
    ↓
┌─────────────────────────────────┐
│  引导层：BOOTSTRAP              │
│  → 加载身份/行为/记忆/环境       │
│  → 检查 RAGFlow 可用性           │
└─────────────────────────────────┘
    ↓
┌─────────────────────────────────┐
│  身份层 + 行为层 + 记忆层        │
│  → 理解问题 + 确定回答策略      │
└─────────────────────────────────┘
    ↓
    ┌─ 普通问题 ─→ 直接回答 ─┐
    ↓                        ↓
┌──────────────────┐    ┌──────────────────┐
│   需要检索？      │是  │   检索增强模式    │
│   → 是          │───→│   → 调用 RAGFlow  │
│   → 否          │    │   → 获取候选片段 │
└──────────────────┘    │   → 解释/过滤    │
                        │   → 生成回答     │
                        └──────────────────┘
    ↓
┌─────────────────────────────────┐
│  记忆层：判断是否沉淀            │
│  → 是 → 写入 MEMORY.md         │
│  → 否 → 写入 memory/YYYY-MM-DD │
└─────────────────────────────────┘
    ↓
用户收到回答
```

## 四、信息流动总览

### 启动信息流

```
BOOTSTRAP
 → 加载身份层（SOUL → IDENTITY → USER）
 → 加载行为层（AGENTS → HEARTBEAT）
 → 加载记忆层（MEMORY → 当日日志）
 → 加载环境层（TOOLS）
 → 检查 RAGFlow 可用性
 → 建立当前任务上下文
 → 进入会话
```

### 问题处理信息流

```
用户问题
 → 判断任务类型
 → 检查是否需要检索增强
   → 若需要 → 调用 RAGFlow → 获取候选片段
   → 若不需要 → 直接进入回答生成
 → 结合身份/行为/记忆进行解释
 → 生成结构化输出
 → 判断是否需要沉淀记忆
 → 写入日志
```

### 记忆沉淀信息流

```
会话结果
 → 判断是否长期有效
   → 长期有效 → 写入 MEMORY.md
   → 阶段性上下文 → 写入 memory/YYYY-MM-DD.md
   → 高频复用 → 更新 FAQ / TERMS
```

## 五、核心设计原则

| 原则 | 说明 |
|------|------|
| 分层解耦 | 不同问题由不同层负责，便于维护和替换 |
| 本地约束优先 | 项目内文件定义优先于模型自由发挥，降低角色漂移和幻觉 |
| 检索增强而非检索替代 | RAGFlow 负责补充知识，不替代 Agent 的判断与表达 |
| 长期与短期记忆分离 | 稳定信息与每日流水分开，避免记忆污染 |
| 治理优于堆功能 | 不仅要能回答，更要能持续稳定地回答 |

## 六、典型工作模式

| 模式 | 适用场景 | 依赖层 |
|------|----------|--------|
| 普通回答模式 | 寒暄、通用说明、非项目知识问题 | 身份层 + 行为层 + 当前上下文 |
| 检索增强模式 | 项目问答、流程确认、配置说明、版本差异、高风险问题 | 身份层 + 行为层 + 记忆层 + 环境层 + RAGFlow |
| 规范生成模式 | 生成 Markdown/SOP/FAQ/角色画像 | 身份层 + 行为层 + 记忆层 + RAGFlow + 术语/FAQ |
| 降级模式 | RAGFlow 不可用、检索失败、依据不足 | 本地 Markdown 文件 + 当前上下文 + 长期记忆 |
